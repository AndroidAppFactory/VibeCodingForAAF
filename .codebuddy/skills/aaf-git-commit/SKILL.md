# AAF Git 提交助手

## 概述

**AI 辅助完成** Git 代码提交，通过以下流程：
1. 🔍 检查变更状态
2. 📊 分析变更内容
3. 📝 生成规范提交方案
4. ❓ 展示方案并等待授权
5. ✅ 执行提交（授权后）
6. 🚀 询问是否推送

## 触发关键词

- "准备提交" / "准备提价"
- "提交代码" / "帮我提交"
- "提交信息" / "提交文案" / "提交规范"
- "commit" / "commit message"
- "总结提交" / "总结一下XX的提交"
- "生成提交信息"
- "怎么提交"

## 核心原则

### 规范引用

提交信息格式严格遵循 `aaf_git.mdc` 规范（使用 `read_rules` 工具读取 `aaf_git` 获取详细规范）。

关键要点：
- **格式**：`<type>(<scope>): <subject>`
- **Subject**：中文、动词开头、不超过 50 字符、不加句号
- **Type**：feat / fix / docs / style / refactor / perf / test / build / ci / chore / revert
- **Scope**：模块名或功能名

### 授权规则（强制）

- ❌ **绝对禁止**未经授权执行 `git commit` 或 `git push`
- ✅ 每次提交都需要独立授权，不可复用
- ✅ 授权词汇："可以"、"执行"、"提交吧"、"1"（选项）等
- ✅ 发现新问题需重新展示方案并获取授权

## 任务进度展示（必须）

**AI 必须使用 `todo_write` 工具展示提交进度**。

### 初始化任务列表

```json
[
  {"id": "1", "status": "in_progress", "content": "检查变更状态"},
  {"id": "2", "status": "pending", "content": "分析变更内容"},
  {"id": "3", "status": "pending", "content": "生成提交方案"},
  {"id": "4", "status": "pending", "content": "等待用户授权"},
  {"id": "5", "status": "pending", "content": "执行提交"}
]
```

### 状态更新规则

- 每完成一个阶段，更新对应任务为 `completed`，下一个任务为 `in_progress`
- 使用 `merge=true` 只更新变化的任务
- 阶段 4（等待授权）保持 `in_progress` 直到用户回复

## 工作流程

```
用户请求提交
    ↓
【创建任务列表】- 使用 todo_write 显示 5 个阶段
    ↓
【阶段 1：检查变更】✅ → 更新进度
AI 执行：
├─ 智能定位项目（见项目定位策略）
├─ git status --short
├─ git diff --stat
└─ 无变更 → 告知用户并结束
    ↓
【阶段 2：分析变更】✅ → 更新进度
AI 执行：
├─ git diff 查看具体变更内容
├─ 对大量变更按模块/功能分组查看
├─ 识别变更类型（新增/修改/删除）
├─ 判断变更性质（feat/fix/refactor 等）
└─ 确定影响范围（模块/文件/功能）
    ↓
【阶段 3：生成提交方案】✅ → 更新进度
AI 执行：
├─ 判断是单次提交还是拆分多次
├─ 为每个提交选择合适的 Type 和 Scope
├─ 生成规范的 Subject 和 Body
├─ 展示完整的提交方案表格
└─ 提供选项（执行/取消/修改）
    ↓
【阶段 4：等待授权】⏳ 保持 in_progress
AI 展示：
├─ 变更分析表格
├─ 提交方案（git add + git commit 命令）
├─ 询问："是否执行？"
└─ 等待用户回复
    ↓
【阶段 5：执行提交】✅ → 全部完成
用户授权后，AI 执行：
├─ 按方案执行 git add + git commit
├─ 反馈每个提交的结果
├─ 询问是否推送到远程
└─ 推送需要二次授权
```

## 项目定位策略

### 智能识别当前项目

AI 需要判断用户想在哪个项目提交，按优先级：

1. **上下文推断** - 根据对话历史判断用户正在操作的项目
2. **用户明确指定** - 用户说"提交 XX 项目"
3. **打开的文件** - 根据 IDE 当前打开的文件路径判断项目
4. **多项目检查** - 如果无法确定，检查常用项目的变更状态

## 提交方案生成规则

### 单次提交 vs 多次提交

**单次提交**：变更集中在同一模块/同一目的
```
git add <files>
git commit -m "type(scope): subject"
```

**多次提交**：变更涉及多个模块/多个目的
```
【提交 1】模块 A 相关
git add <files_a>
git commit -m "type(scopeA): subject_a"

【提交 2】模块 B 相关
git add <files_b>
git commit -m "type(scopeB): subject_b"
```

### 拆分原则

- 不同模块的变更拆分为不同提交
- 不同目的的变更拆分（如 feat + fix 分开）
- 配置变更和代码变更可以分开
- 单文件小变更可合并为一个提交

### 展示格式

```
📋 变更分析

| 文件 | 变更 | 说明 |
|------|------|------|
| path/to/file1 | 修改 | 具体说明 |
| path/to/file2 | 新增 | 具体说明 |

💡 提交方案

【提交 1】
git add <files>
git commit -m "type(scope): subject

- body line 1
- body line 2"

是否执行？
1. 执行所有提交
2. 取消
3. 修改提交方案
```

## 场景处理

### 场景 1：用户说"准备提交" / "提交代码"

```
完整流程：
1. 检查变更 (git status + git diff)
2. 分析变更内容
3. 生成提交方案
4. 展示方案，询问"是否执行？"
5. 等待授权
6. 授权后执行
```

### 场景 2：用户说"总结一下XX的提交"

```
⚠️ 重要："提交"指的是【待提交的变更】，不是【已提交的历史】

流程：
1. git status 查看待提交文件
2. git diff 查看具体变更
3. 分析每个文件的变更目的
4. 生成变更总结报告
5. 生成提交建议
6. 可询问是否执行，但需授权
```

### 场景 3：用户说"帮我提交" / "你帮我执行"

```
流程：
1. 检查变更
2. 生成方案
3. 展示："建议如下提交，是否执行？"
4. 等待授权（即使用户说了"帮我"，仍需确认方案）
5. 授权后执行
```

### 场景 4：提交后询问推送

```
提交成功后：
AI 展示：
  ✓ 提交成功！[hash] type(scope): subject
  
  是否推送到远程？
  1. 执行推送
  2. 取消

推送也需要独立授权。
```

### 场景 5：多个项目有变更

```
流程：
1. 逐个检查各项目变更
2. 分别生成提交方案
3. 统一展示所有项目的方案
4. 询问授权（可一次授权全部，也可逐个）
5. 按授权执行
```

### 场景 6：用户说"看看有什么变更"

```
只读操作：
1. 执行 git status 和 git diff
2. 展示变更内容
3. 不询问是否提交（用户只是想看）
```

## 安全检查

### 提交前必查

- ✅ 无敏感信息（API Key、密码、Token）
- ✅ 无自动生成文件（build/、.idea/、*.class、*.apk）
- ✅ 无临时文件（*.tmp、*.log、*.swp）
- ✅ 变更内容都是有意的

### 授权确认清单

```
在执行 git commit/push 前必须确认：
□ 已向用户展示完整提交方案
□ 已明确询问"是否执行？"
□ 用户已回复授权词汇
□ 授权针对当前提交（非复用）
```

## AI 执行检查清单

### 变更检查
- [ ] 定位正确的项目目录
- [ ] 执行 git status 检查变更
- [ ] 执行 git diff 查看内容
- [ ] 确认无敏感信息

### 方案生成
- [ ] 正确选择 Type
- [ ] 准确确定 Scope
- [ ] Subject 中文、动词开头、≤50 字符
- [ ] 必要时添加 Body
- [ ] 合理拆分多次提交

### 授权与执行
- [ ] 展示完整方案
- [ ] 明确询问"是否执行？"
- [ ] 等待用户授权
- [ ] 授权后才执行
- [ ] 执行后反馈结果
- [ ] 询问是否推送（独立授权）

## 注意事项

- ❌ 未经授权不执行任何写操作（commit/push）
- ❌ 不复用之前的授权
- ❌ "总结提交"不是查看 git log，而是分析待提交变更
- ✅ 只读操作（status/diff/log）随时可执行
- ✅ 每次提交独立授权
- ✅ 推送需要二次独立授权
- ✅ 多项目场景逐个处理或统一展示
